# Descriptive Profiling Performance

## Scope

This document covers uploaded CSV datasets and saved Data Views materialized
from them. The profiling contract remains role-aware and produces the same UI
sections, but computation has moved from React to a backend analytical worker.

## Previous Bottlenecks

The former path was bounded at 50,000 rows and had several scaling failures:

1. FastAPI read the complete multipart upload into a Python `bytes` value.
2. Preview parsed the complete CSV into nested Python lists before applying its
   return limit.
3. Up to 50,000 full records were serialized to JSON and transferred to React.
4. React repeatedly sorted, grouped, and copied those records for each column,
   relationship, density curve, and segment pair.
5. Raising the limit would grow browser memory, transfer size, and main-thread
   work without providing a viable path to tens of millions of rows.

## Current Execution Path

1. Upload is copied from Starlette's spooled file to repository storage in 1 MiB
   chunks.
2. A streaming CSV pass determines row count and source types without retaining
   rows.
3. Dataset selection reads lightweight schema context only.
4. `Run profiling` queues a Celery task and the frontend polls its status.
5. The worker creates or reuses `dataset.mlapp.parquet` and executes DuckDB
   aggregates over all rows.
6. Redis and the API return only column profiles, relationship summaries,
   bounded graphic data, segment results, and quality notes.

Core statistics are full-data calculations. The configured graphic source-point
limit only bounds scatterplot observations. Histograms and density-like curves
are built from full-data bin counts.

DuckDB defaults to four execution threads, a 1 GB per-connection memory limit,
disk spilling, disabled insertion-order preservation, and a dataset-local
temporary directory. Interactive visualization rendering is limited to two
concurrent queries per API process. Celery defaults to two worker processes,
late acknowledgements, and a prefetch
multiplier of one. These values favor predictable memory use over maximum
concurrent throughput on the local Docker stack. They can be tuned with
`DUCKDB_THREADS` (or the legacy `DESCRIPTIVE_PROFILE_DUCKDB_THREADS`),
`DUCKDB_MEMORY_LIMIT`,
`VISUALIZATION_MAX_CONCURRENCY`, `PROFILE_WORKER_CONCURRENCY`, and
`DESCRIPTIVE_PROFILE_RESULT_EXPIRES_SECONDS`.

## Reproducible Benchmark

Run:

```powershell
docker exec ml-app-api-1 python tests/benchmark_full_profile.py --rows 10000000
```

Measured in the local Docker environment on 2026-06-20:

| Rows | Columns | CSV | Parquet | Full profile |
| ---: | ---: | ---: | ---: | ---: |
| 1,000,000 | 6 | 29.06 MiB | 1.03 MiB | 2.296 s |
| 10,000,000 | 6 | 300.14 MiB | 9.95 MiB | 17.205 s |

The timer includes first-run Parquet creation plus univariate profiles,
target-feature relations, graphic aggregates, and multivariate segment scanning.
Synthetic data compresses unusually well, so these numbers are regression
anchors rather than universal capacity promises. Width, string cardinality,
exact distinct counts, medians, storage speed, and concurrent jobs can dominate
real workloads.

Profiling range options are applied before expensive work. Disabling Univariate
profiles skips per-column medians, modes, examples, and histograms except for
the active comparison context. `Max target relation features` stops relation
construction after the requested number of eligible source-order columns rather
than calculating and discarding additional results.

## Complexity And Limits

- Memory no longer scales with `rows * columns` in Python or the browser.
- Most aggregates are vectorized linear scans with Parquet column pruning.
- Exact median and exact distinct count remain the most expensive operations for
  very wide or high-cardinality datasets.
- Parquet materialization requires temporary disk capacity in addition to the
  source CSV.
- DuckDB is a single-node engine. Tens of millions of rows are a realistic local
  target, but multi-terabyte or distributed datasets should use a remote query
  engine such as Spark or Trino behind the same compact profile response.
- Saved SQL and Browser Data Views inherit the same columnar guarantees. Their
  transformations are pushed into DuckDB and cached as definition-versioned
  Parquet relations; nested views are resolved recursively with bounded depth.

## Regression Coverage

`backend/tests/test_full_profile.py` verifies that:

- rows beyond the former 50,000-row limit affect statistics,
- schema selection does not trigger Parquet materialization,
- numeric/categorical relations, graphics, and segments are generated by the
  backend profile.

`backend/tests/test_visualizations.py` verifies full-row visualization
aggregates, density binning, group selection, SQL and Browser Data Views,
nested views, independent scatter epsilon bins, bounded density-first output,
per-group trend fitting and metadata, typed response aliases, and continuation
from a view into descriptive profiling.

`backend/tests/benchmark_full_profile.py` is intentionally manual so routine
test runs remain fast while performance can be measured at realistic scales.
